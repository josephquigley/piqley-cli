Rule Editor Architecture — Three Approaches

How to split responsibilities between PiqleyCore and the CLI

A

Thin Core — Validate & Serialize Only

PiqleyCore adds a RuleBuilder that validates individual fields (is this pattern syntactically valid? is this action known?) and handles stage file I/O. The CLI wizard owns all workflow logic — step ordering, field selection, action composition.

Pros

  • Minimal Core changes — mostly validation helpers
  • CLI has full control over UX flow
  • Fast to ship

Cons

  • GUI must reimplement rule assembly logic
  • Risk of validation drift between clients
B

Builder Pattern in Core — Guided Assembly

PiqleyCore provides a RuleBuilder with a fluent API that enforces valid construction: builder.match(field:pattern:).addEmit(...).addWrite(...).build(). Core ensures you can't build an invalid rule. The CLI wizard maps user input to builder calls. A GUI would do the same.

Pros

  • Invalid rules are impossible to construct
  • Both CLI and GUI share assembly logic
  • Clean API boundary — Core says what's valid, clients drive UX

Cons

  • Slightly more Core surface area
  • Builder must be flexible enough for both clients
C

Context Object in Core — Client Injects, Core Guides

PiqleyCore provides RuleEditingContext — the client injects available fields, installed plugins, and existing rules. Core then exposes what's possible given that context: valid fields to match, valid patterns for a field, valid actions for an emit/write. The CLI queries the context at each wizard step.

Pros

  • Core is the single source of truth for "what's valid here"
  • Clients stay thin — just render choices Core provides
  • Field discovery logic injected, not owned by Core
  • GUI gets the same context-aware validation for free

Cons

  • Most upfront design work
  • Context object must model the full editing domain

⭐ Recommendation: C — Context Object

This matches what we discussed: Core can't discover anything, but given injected context it can tell clients exactly what's valid at each step. The CLI wizard becomes a thin loop — ask Core "what are my options?", show them to the user, feed the choice back. A future GUI does the same thing with dropdowns instead of prompts. The upfront cost is modest since the domain is well-scoped (match → emit → write).
